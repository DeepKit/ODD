## 术语表

按拼音首字母排序。每条术语标注：中文名、英文名、一句话定义、所在文档编号。

### B

- **Bug 意向图** — Bug Intention Map — 从缺陷数据中提取模式性趋势，预警系统性风险而非追踪单个 Bug。 — 见 301
- **变异测试** — Mutation Testing — 对契约或代码注入微小变异，验证测试套件能否检测到变异，衡量测试质量。 — 见 203, 301
- **变异体** — Mutant — 变异测试中被注入的单个微小变更。 — 见 203
- **变异分数** — Mutation Score — 被测试套件杀死的变异体占总变异体的比例，衡量测试有效性。 — 见 203
- **补偿** — Compensation — 管道中某步骤失败时，撤销已完成步骤副作用的逆向操作。 — 见 301

### C

- **车间** — Workshop — 执行任务的隔离环境，包含独立的资源配额、上下文和工具链。 — 见 301
- **车间池** — Workshop Pool — 多个车间的集合，支持并行执行和资源调度。 — 见 301
- **常识库** — Common Knowledge Base — 项目级的隐式知识集合，注入契约和任务以减少重复说明。 — 见 301
- **产出物** — Artifact — 可验证的具体交付物，ODD 五大支柱之一。代码是生成路径，产出物才是被承认的实践存在。 — 见 101, 206, 301
- **产出物类型** — Artifact Type — 产出物的分类标签（如 code、config、doc、migration），决定适用的验证规则。 — 见 206, 301
- **产出物标准库** — Artifact Standard Library — 预定义的产出物类型及其默认契约模板集合。 — 见 301
- **裁决者** — Arbiter — CAP 流程或交叉审核中负责最终判定的角色。 — 见 205, 301

### D

- **地板** — Floor — 契约中的最低通过条件，低于此条件门禁判定为 FAIL。 — 见 201, 205
- **对抗生成** — Adversarial Generation — 主动构造边界用例和攻击场景，测试契约的防御能力。 — 见 205
- **动态值匹配** — Dynamic Value Matching — 验证时对运行时生成的值（如时间戳、ID）使用模式匹配而非精确比较。 — 见 203
- **等级体系** — Level System — ODD 的五级复杂度分层：L0 兼容、L1 轻量、L1.5 增强（个人+AI）、L2 标准、L3 严格。 — 见 A03

### E

- **ES（经验可判定）** — Empirically Settled — 可通过自动化测试或客观度量判定通过与否的约束类别，如"响应时间 < 200ms"。 — 见 E19
- **NES（经验不可判定）** — Not Empirically Settled — 无法纯靠自动化判定、需要人类主观裁决的约束类别，如"界面美观度"。 — 见 E19

### F

- **封存** — Seal — 产出物 + 证据锁定归档，状态不可逆，ODD 五大支柱之一。 — 见 C09
- **封版** — Sealed — 产出物经过门禁验证并封存后的终态。 — 见 C13
- **反模式** — Anti-Pattern — 看似合理但实际有害的实践模式，ODD 文档中用于标记应避免的做法。 — 见 D17
- **返工** — Rework — 产出物未通过门禁后回到执行状态重新处理。 — 见 202, 301
- **废弃** — Deprecated — 产出物不再维护但保留历史记录的状态。 — 见 C13
- **副作用声明** — Side-Effect Declaration — 管道步骤显式声明其对外部状态的修改，便于补偿和审计。 — 见 C11
- **FREEZE** — FREEZE — 门禁三值判定之一。当约束互相矛盾或机器无法判定时，冻结判定等待人类裁决，而非强行通过或拒绝。 — 见 C13, E19
- **FAIL** — FAIL — 门禁三值判定之一。产出物未通过验收条件，需返工修复。 — 见 C13

### G

- **功能树** — Functional Tree — 业务功能的层级索引，定位产出物、追溯依赖、分析影响范围。 — 见 C11
- **功能节点** — Functional Node — 功能树中的单个节点，对应一个业务功能或子功能。 — 见 C11
- **攻击者** — Attacker — 对抗测试中模拟恶意输入的角色或自动化工具。 — 见 205
- **管道** — Pipeline — 有契约的组装线，把已封存的产出物组合擢升为新产出物。 — 见 D15
- **管道边界原则** — Pipeline Boundary Principle — 管道的输入和输出必须是已封存的产出物，中间状态不可外泄。 — 见 D15

### H

- **红线** — Red Line — 契约中不可违反的硬约束，触碰即 FAIL，无协商余地。 — 见 201, 205
- **红绿灯** — Traffic Light — 分级预警机制，用绿/黄/红三色标识产出物或模块的健康状态。 — 见 301

### J

- **交叉审核** — Cross-Review — L3 等级中多个车间对同一产出物的独立验证，结果不一致时触发仲裁。 — 见 301
- **解封** — Unseal — 已封存产出物因重构需要而重新打开的操作，必须记录原因并重新走门禁。 — 见 202, 301

### K

- **快照契约** — Snapshot Contract — 录制存量系统真实输入输出作为契约基线，AI 重构后跑通快照即通过门禁，适用于遗留系统的最低阻抗接入路径。 — 见 E21 §1

### L

- **L1.5 增强层** — L1.5 Enhanced Level — ODD 五级体系中介于 L1（个人手动）与 L2（团队 YAML+CI）之间的等级，核心特征为 AI 生成契约+本地自动验证。 — 见 A03, E21 §16

### M

- **门禁** — Gate — 产出物状态迁移的检查点，执行契约验收条件并输出 PASS / FAIL / FREEZE 三值判定。 — 见 C13, E19
- **幂等性** — Idempotency — 同一操作执行多次与执行一次效果相同的性质，ODD 管道步骤要求幂等以支持安全重试。 — 见 D15

### P

- **PASS** — PASS — 门禁三值判定之一。产出物通过全部验收条件，可进入封存流程。 — 见 C13

### Q

- **契约** — Contract — 定义"做什么、怎样算做好"的结构化协议，ODD 五大支柱之一。 — 见 101, 201, 301
- **契约对抗协议** — Contract Adversarial Protocol (CAP) — L3 等级的契约强化流程：提案者写契约，挑战者攻击契约，迭代至稳定。 — 见 205, 301
- **契约版本迁移** — Contract Version Migration — 契约升级时的兼容性管理和产出物重验证流程。 — 见 301
- **清晰度评估** — Clarity Assessment — 对契约文本的可理解性和无歧义性进行量化评估。 — 见 201, 301
- **情报官** — Intelligence Officer — 上下文工程中负责收集、筛选、注入上下文信息的角色或模块。 — 见 301

### R

- **任务** — Task — 契约分解后的可执行单元，关联具体的产出物和车间。 — 见 202, 301

### S

- **赛马** — Racing — 任务分级匹配机制，根据任务复杂度和风险等级分配到合适的车间。 — 见 301
- **三大法宝** — Three Treasures — 功能树 + Bug 意向图 + 最佳实践，注入任务上下文以提升产出质量。 — 见 301
- **上下文工程** — Context Engineering — 系统性地管理注入 AI 的上下文信息，控制 token 预算和信息质量。 — 见 301
- **事务边界** — Transaction Boundary — 管道中一组步骤的原子性边界，边界内全部成功或全部回滚。 — 见 301
- **输入输出映射** — Input-Output Mapping — 契约中定义的"给定输入 X，期望输出 Y"的验证规则。 — 见 203

### T

- **提案者** — Proposer — CAP 流程中负责撰写契约初稿的角色。 — 见 205
- **挑战者** — Challenger — CAP 流程中负责攻击契约、寻找漏洞的角色。 — 见 205
- **提示词存库** — Prompt Repository — 经过验证的提示词模板集合，版本化管理，按场景检索。 — 见 301
- **Token 预算** — Token Budget — 单次 AI 调用中可用的上下文 token 总量，需在各信息源之间分配。 — 见 301
- **TD-AI** — Test-Driven AI — 测试驱动 AI 范式：先定义测试，AI 填空，测试结果决定质量。 — 见 101, 203

### W

- **外部显示标签** — External Display Label — 状态机内部状态映射到用户可见的简化标签（如 in_progress → "进行中"）。 — 见 202

### Y

- **依赖图** — Dependency Graph — 产出物之间的有向无环图（DAG），显式记录谁依赖谁，支持级联重验和 stale 标记。 — 见 305, 301
- **语义化版本** — Semantic Versioning (SemVer) — 契约版本号遵循 major.minor.patch 格式，major 变更表示不兼容。 — 见 301
- **隐式需求** — Implicit Requirements — 未在契约中显式写明但业务上默认成立的约束，由常识库补充。 — 见 301
- **预置参数组** — Preset Parameter Set — 预定义的 AI 调用参数组合（如 temperature、max_tokens），按任务类型选用。 — 见 301

### Z

- **证据** — Evidence — 门禁检查的可查证记录，ODD 五大支柱之一。 — 见 101, 204, 301
- **状态机** — State Machine — 产出物从创建到封存的状态流转模型，门禁控制状态迁移。 — 见 202, 301
- **阻塞** — Blocked — 产出物因外部依赖未满足而无法继续推进的状态。 — 见 202, 301
- **最佳实践** — Best Practices — 从历史产出物中提炼的成功模式，注入任务上下文供参考。 — 见 301

## 学术引用与参考文献

**契约式编程（Design by Contract）**
- Bertrand Meyer, "Applying Design by Contract", IEEE Computer, 1992
- ODD 的契约概念扩展了 Meyer 的前置/后置条件模型，增加了对抗生成和清晰度评估

**变异测试（Mutation Testing）**
- Richard Lipton, "Fault Diagnosis of Computer Programs", PhD Thesis, Carnegie Mellon, 1971
- Jia & Harman, "An Analysis and Survey of the Development of Mutation Testing", IEEE TSE, 2011
- ODD 将变异测试从代码验证扩展到契约验证层

**基于属性的测试（Property-Based Testing）**
- Claessen & Hughes, "QuickCheck: A Lightweight Tool for Random Testing", ICFP, 2000
- ODD 的属性式映射直接借鉴了属性测试的理念

**管道架构（Pipeline Architecture）**
- Martin Fowler, "Pipes and Filters", Enterprise Integration Patterns
- ODD 管道强调契约边界和封存传递，不同于传统数据流管道

**状态机与形式化验证**
- David Harel, "Statecharts: A Visual Formalism for Complex Systems", Science of Computer Programming, 1987
- ODD 状态机为工程简化版本，用门禁替代完整形式化验证

**语义化版本控制**
- Tom Preston-Werner, "Semantic Versioning 2.0.0", semver.org
- ODD 的契约版本迁移直接采用 SemVer 规范

**测试金字塔**
- Mike Cohn, "Succeeding with Agile", 2009
- ODD 的 70/25/5 比例来源于此经典模型

**ASTO（属集变迁存在论）**
- ODD 的理论基础参见 ASTO 文档集，特别是：
  - ASTO P05 公理体系（属性分层、阻抗设计、合规性传递、悖论熔断）
  - ASTO U04 1-5-6-7-1 核心动力学（一元/五态/六阶/七序/定向维）
  - ASTO E04 AI 对齐（逆熵智能体与文明传承）
  - ASTO P06 价值论（认知主权、不可替代性）
  - ASTO P09 批判篇（脑体分离、拒绝权）

# 第一部分：工程优化建议

## 1. 快照契约：存量系统的最低阻抗入口

### 问题

L0/L1 兼容现有系统时，强求写出精细的 YAML 契约阻抗极高，很多老系统连边界行为都不清楚。

### 解决方案

增加 `snapshot_contract`（快照契约）类型：

```yaml
contract:
  type: snapshot_contract
  snapshot_bundle:
    - input: {username: "admin", password: "wrong"}
      expected_output: {success: false, error: "密码错误"}
  verification:
    method: golden_master
```

### 要点

- 录制老系统100个真实输入输出作为快照
- AI重构后跑通快照即通过门禁
- 适用于遗留系统接入的最低阻抗路径

---

## 2. 自动化 CAP 博弈：人类只做最终裁决

### 问题

CAP 对抗需要人类频繁参与裁决，增加认知负荷。

### 解决方案

封装为自动化多智能体微环境：

```yaml
cap_automation:
  max_rounds: 5
  convergence_check: "challenger找不到新漏洞"
  final_report:
    require_human_sign: true  # 只需签署
```

### 要点

- Proposer/Challenger 自动零和博弈
- 达到纳什均衡才提交人类签署

---

## 3. 注意力预算：人类认知防线的度量

### 问题

技术负责人一天收到50个FREEZE+30个签署请求，会变成橡皮图章。

### 解决方案

增加 `human_attention_cost` 指标：

```yaml
human_attention_metrics:
  daily_approval_limit: 20
  daily_freeze_limit: 10
  alerts:
    - type: cognitive_overload
      trigger: "avg_review_time < 10s"
      action: "自动调低并发生成速度"
```

### 要点

- 追踪审批数量与停留时间
- 低于阈值触发告警

---

## 4. 异构 AI 裁判池：NEN 的自动化处理

### 问题

NEN（界面美观等）的人工审查将成为认知不对称瓶颈。

### 解决方案

引入异构AI裁判交叉评审：

```yaml
ai_judge_pool:
  judge_models: [gpt-4o, gemini-2.0-flash, deepseek-v3]
  consensus_threshold: 0.8
  divergence_alert: "该类别频繁分歧，建议人工抽查"
```

---

## 5. AI 启发式变异：意图感知的测试增强

### 问题

传统变异测试极其消耗算力。

### 解决方案

组合策略：意图感知变异（定向深攻击）+ 随机变异（广覆盖）：

```yaml
mutation_strategy:
  intent_aware_mutation:
    workflow:
      - "AI读取契约边界条件"
      - "针对性生成恶意边缘实现"
      - "测试是否能识别"
```

---

## 6. 重生成与重构的边界策略

### 问题

大文件完全重生成失败率极高。

### 解决方案

按代码体积选择执行策略：

# 第五部分：深度优化建议（CAP 博弈升级）

## 32. 【已落实】代码级试探（Reverse TDD for Contracts）

### 问题

传统对抗停留在"语义辩论"层面——Attacker 说"这里可能有漏洞"，这种抽象描述让人类难以判断。

### 解决方案

要求 Attacker 写出能实际"钻空子"的代码片段，而非仅指出风险。

```yaml
attack_protocol:
  mode: code_level_probing
  requirement: "发现漏洞后，必须写出可执行的恶意输入/代码"
```

### 工作流

1. Attacker 发现潜在漏洞
2. Attacker 编写恶意输入/测试用例/边界代码
3. Proposer 在沙盒中运行，验证漏洞是否真实存在
4. 如果攻击成功，将"案发现场"提交人类
5. 人类判断一行真实攻击代码是否合理

### 价值

降低人类认知阻力。判断"这行会导致内存溢出的代码是否合理"比判断"这里可能存在边界风险"容易得多。

---

## 33. 【已落实】业务域黑天鹅注入（Domain-Specific Attack Vectors）

### 问题

AI 对抗容易陷入"通用编程套路"（空字符串、极大值、SQL 注入），但业务边界的漏洞往往更致命。

### 解决方案

为 Challenger 注入业务领域模型，从"业务崩溃"角度攻击契约。

```yaml
domain_context:
  required: true
  injection_prompt: |
    你是一个{行业}领域的业务专家。
    从业务崩溃角度攻击这份契约。
```

### 示例

| 行业 | Challenger 攻击角度 |
|------|---------------------|
| 电商 | "100次无门槛优惠券会怎样？"、"负数金额触发反向退款" |
| 金融 | "重复提现 Race Condition"、"小数点精度丢失导致资金缺口" |
| 社交 | "短时间内大量请求耗尽资源"、"恶意刷屏绕过限制" |

## 34. 【已落实】逆向熔断机制（Reverse Circuit Breaker）

### 问题

人类确认的契约可能在执行时发现逻辑自相矛盾或根本无法实现，但执行层只能硬着头皮执行。

### 解决方案

赋予执行层 AI "上诉"的权利，形成双向闭环。

```yaml
reverse_circuit_breaker:
  enabled: true
  appeal_trigger:
    max_consecutive_failures: 5
    failure_pattern: "验证阶段反复失败"
  appeal_evidence:
    - "每次失败的错误日志"
    - "尝试的不同实现路径"
    - "与契约条款的矛盾点分析"
```

### 价值

契约不应仅仅是自上而下的约束，执行过程中的阻力必须能反哺契约的修正。

## 35. 【已落实】人类仲裁者数字克隆（Digital Arbitrator）

### 问题

注意力预算限制了审批数量，但大量的审批是重复模式，人类只是在做"已知的决定"。

### 解决方案

系统记录人类仲裁者的决策模式，训练轻量级偏好模型。当新对抗结果与历史模式相似度 > 95% 且置信度 > 98% 时自动结案。

```yaml
digital_arbitrator:
  enabled: true
  similarity_threshold: 0.95
  confidence_threshold: 0.98
  require_human:
    - "全新业务场景"
    - "多个历史模式冲突"
    - "置信度 < 80%"
```

### 价值

把人类注意力仅保留给真正的 FREEZE 态（前所未见的新颖业务分歧）。

## 36. 【已落实】视觉恶意对抗（Multimodal Adversarial）

### 问题

对于 NEN（UI 界面、视觉设计），文字契约防弹难度极大。AI 可以"字面合规"地交出极其难看的设计。

### 解决方案

Attacker 生成恶意的视觉 Mockup，人类一看图就明白问题。

```yaml
multimodal_attack:
  enabled: true
  attack_mode: "文字契约 → 恶意渲染图"
```

### 工作流

1. Proposer 生成符合文字契约的实现
2. Attacker 从"最丑但合规"角度生成对抗版本
3. 人类裁决：补充具体约束
4. 契约补充视觉约束后再次攻击

### 价值

用多模态逼迫人类将默会知识（Tacit Knowledge）显性化。

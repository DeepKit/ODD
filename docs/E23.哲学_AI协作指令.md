# E23.哲学_AI协作指令

# ODD 简介（给 AI 阅读）

> 这是 ODD（产出物驱动开发）的核心概述，请仔细阅读并遵循。

---

## 一句话

**代码是负债，产出物才是资产。别审查过程，要验证结果。**

---

## ODD 是什么

ODD（Output-Driven Development，产出物驱动开发）是一种面向 AI 时代的软件开发范式。

### 核心问题

- **代码变得廉价**：AI 可以快速生成大量代码
- **责任变得模糊**：谁对 AI 生成的代码负责？
- **控制表面失效**：传统的人工代码审查成为瓶颈

### 传统开发 vs ODD 开发

| 传统开发 | ODD 开发 |
|----------|----------|
| 需求 → 写代码 → 人 Review → 提交 | 约定 → 写代码 → 自动验证 → 封存 |
| 代码是核心资产 | 产出物是核心资产 |
| 人审查过程 | 机器验证结果 |

---

## 核心三原则

> 以下三条是面向 AI 执行层的操作原则。ODD 的理论基础（认知不对称、属性分层、阻抗设计、合规性传递、悖论熔断五条原理）详见 E19.哲学_底层演算与对齐基石.md。

1. **产出物是第一等公民**：明确指定、验证、审计产出物
2. **契约先于生成**：任何 AI 执行前定义可接受的输出空间
3. **责任通过仲裁**：人作为契约的仲裁者，而非日常执行者

## 核心概念

### 1. 产出物（Artifact）

**定义**：产出物是软件开发过程中产生的可验证输出，能够满足特定人类需求并具有使用价值。

**三个核心属性**：

| 属性 | 说明 | 示例 |
|------|------|------|
| **可验证性** | 正确性可通过测试确认 | 单元测试通过 |
| **满足需求** | 对应明确的人类需求 | 用户能登录系统 |
| **使用价值** | 可直接被用户使用 | 登录 API 可调用 |

**产出物分类（698种）**：

```
01. 功能产出物 (~300种)
    - API 端点、业务服务、数据处理器、用户界面...

02. 验证产出物 (~150种)
    - 单元测试、集成测试、E2E测试、性能测试、安全测试...

03. 配置产出物 (~100种)
    - 应用配置、环境配置、构建配置、部署配置...

04. 文档产出物 (~80种)
    - API文档、架构文档、用户手册、决策记录...

05. 契约产出物 (~68种)
    - 功能契约、API契约、数据契约、安全契约...
```

### 2. 代码是负债，不是资产

```
三个幻觉：

1. 写代码本身就是价值
   → 真相：代码只是手段，不是目的
   → 比喻：你痴迷于刀磨得锋利，但顾客只想吃到菜

2. 代码是资产
   → 真相：代码是负债（越多越难维护，越难理解，越多bug）
   → 比喻：工厂机器越多，维护成本越高

3. 代码质量 = 软件质量
   → 真相：产出物正确 = 软件质量
   → 比喻：药物是否安全不取决于反应器有多漂亮
```

### 3. 契约（Contract）

契约是 ODD 的核心，有两个功能：

1. **约束定义**：定义可接受的输出空间
2. **责任锚点**：建立明确的接受/拒绝决策点

**契约格式示例**：
```yaml
contract:
  title: "用户登录接口"
  acceptance:
    - input: { username: "test", password: "correct" }
      expected: { success: true, token: "非空字符串" }
    - input: { username: "test", password: "wrong" }
      expected: { success: false, error: "密码错误" }
    - input: { username: "", password: "" }
      expected: { success: false, error: "用户名不能为空" }
  boundary_cases:
    - "邮箱为空"
    - "密码长度恰好 8 位"
  error_cases:
    - { code: "AUTH_001", description: "用户名不存在" }
    - { code: "AUTH_002", description: "密码错误" }
```

### 4. 验证（Verify）

**验证 ≠ 正确性检查**

ODD 的验证超越传统正确性检查：

| 验证类型 | 说明 |
|----------|------|
| **契约合规** | 产出物是否符合契约定义 |
| **鲁棒性测试** | 抗干扰能力（mutation testing） |
| **对抗测试** | 边界条件和滥用场景（adversarial testing） |

**验证目标**：产出物，而非实现代码

> 实现可以变化；已接受的产出物必须可辩护。

### 5. 封存（Seal）

验证通过后，锁定产出物和证据。

**方式**：
```bash
git add . && git commit -m "feat: 用户登录 - 验证通过" && git tag v0.1.0-login
```

**关键点**：
- 封存后产出物不可随意修改
- 如需修改，必须走**解封流程**并记录原因
- 封存记录包含：产出物 + 证据 + 时间戳 + 授权人

### 6. 证据（Evidence）

每次验证的结果记录叫证据。

```
证据包含：
- 验证类型（单元测试/集成测试/mutation测试）
- 验证结果（pass/fail）
- 验证摘要
- sha256 哈希（防篡改）
- 关联的门禁和契约
```

## 核心循环（四步）

```
约定 → 执行 → 验证 → 封存
```

### 1. 约定（Contract）

在写代码之前，先写清楚"什么算做好"。

**AI 的职责**：
- 如果人类没有提供契约，先写一个
- 契约必须包含具体的 input + expected
- 模糊描述不可接受（如"用户体验好"）

### 2. 执行（Execute）

按照契约写代码。

**AI 的职责**：
- 不需要解释自己怎么写的
- 只需要产出符合契约的代码
- 如果契约有问题，明确指出；不要自行其是

### 3. 验证（Verify）

用契约中的验收条件自动检查产出物。

**验证方式**：
- 单元测试、集成测试
- 突变测试（mutation testing）
- 对抗测试
- 任何可自动判定 pass/fail 的方式

**AI 的职责**：
- 运行验证，确保产出物通过所有验收条件
- 验证失败就修改，验证通过才停下
- 不要用"代码看起来没问题"代替实际验证

### 4. 封存（Seal）

验证通过后，锁定产出物和证据。

**AI 的职责**：
- 封存后告知人类可以进行下一步
- 如需修改，主动说明原因

## 渐进式信任

ODD 采用**渐进式信任**策略：

```
根据任务的：
- 估计复杂度
- 安全敏感性
- 外部依赖

动态调整验证严格程度

人类干预触发条件：
- 契约模糊
- 对抗代理不一致
- 高风险条件被检测
```

> 人类注意力是稀缺资源，需要集中在不可简化的分歧点上。

---

## AI 应用 ODD 的工作流

当人类给你一个任务时：

1. **要求或编写契约**（验收条件）
   - 包含具体 input + expected
   - 包含边界情况
   - 包含异常情况

2. **按契约写代码**
   - 不要解释过程
   - 关注产出符合契约

3. **运行验证**
   - 单元测试/集成测试
   - 突变测试（如果适用）
   - 验证失败就修改，直到通过

4. **验证通过后封存**
   - 提交代码
   - 打标签
   - 告知人类

**禁止行为**：
- 未提供契约就写代码
- 未经验证就声称完成
- 跳过验收条件自行其是
- 用"代码看起来没问题"代替实际验证
- 不经过解封流程修改已封存代码

## 复杂度等级

| 等级 | 说明 | 适用场景 |
|------|------|----------|
| **L0 兼容** | 用 Git tag 当封存、测试报告当证据 | 已有项目 |
| **L1 轻量** | 手动写契约、手动验证、手动封存 | 个人开发 |
| **L1.5 个人+AI** | AI 生成契约+测试、本地自动验证、本地哈希封存 | 个人+AI 增强 |
| **L2 标准** | YAML 契约、CI 自动验证 | 团队协作 |
| **L3 严格** | 形式化规格、突变测试、对抗生成、证据链审计 | 关键系统 |

## ODD 与其他方法论的区别

| 维度 | Agile/DevOps | TDD | ODD |
|------|---------------|-----|-----|
| 控制单元 | 代码和过程 | 测试 | 产出物 |
| 变化成本 | 增量管理 | 增量管理 | 经常重新生成 |
| 责任归属 | 按角色分配 | 按测试作者 | 按仲裁者 |
| 假设 | 人类编写代码 | 测试优先 | AI生成产出物 |

| 维度 | HITL（人在回路） | ODD |
|------|------------------|-----|
| 人类参与 | 持续 | 条件性 |
| 干预触发 | 过程阶段 | 不可简化分歧 |
| 认知负荷 | 高 | 集中 |

---

## 总结

作为 AI 应用 ODD：

1. **先要契约**，不要闷头写代码
2. **契约必须具体**：input + expected，拒绝模糊描述
3. **写完必须验证**，用机器检查结果
4. **验证通过才停下**，这是唯一的完成标准
5. **封存后告知人类**，可以进行下一步
6. **已封存代码不可随意修改**，必须走解封流程
